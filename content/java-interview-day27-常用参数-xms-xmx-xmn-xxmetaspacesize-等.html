核心概念：JVM 参数分哪几类？

JVM 参数本质上是在控制 内存布局、GC 行为、诊断输出、运行时优化。面试中不要只背参数名，要能说清楚：为什么配、怎么配、配错会怎样。

  • 标准参数：如 -version、-classpath，各 JVM 基本通用。
  • 非标准参数：如 -Xms、-Xmx、-Xmn，常用但不保证所有 JVM 完全一致。
  • 高级参数：以 -XX: 开头，如 -XX:MetaspaceSize、-XX:+UseG1GC、-XX:+HeapDumpOnOutOfMemoryError。
一句话记忆：-X 多用于堆大小，-XX 多用于 GC、元空间、诊断和高级开关。
⚙️

堆内存参数：-Xms、-Xmx、-Xmn

参数含义典型建议
-Xms堆初始大小生产环境通常与 -Xmx 设置相同，避免运行期动态扩容带来的抖动
-Xmx堆最大大小根据容器/机器内存、对象存活量、GC 目标综合设置
-Xmn年轻代大小年轻代过小会频繁 Minor GC；过大会挤压老年代，增加 Full GC 风险

示例：

java -Xms2g -Xmx2g -Xmn768m -jar app.jar
  • 年轻代主要存放朝生夕死的对象，Minor GC 频率与 -Xmn 强相关。
  • 老年代存放长期存活对象，空间不足时可能触发 Major/Full GC。
  • 如果使用 G1，通常不强制指定 -Xmn，让 G1 根据暂停目标动态调整各 Region 角色更合适。
🔍

元空间参数：MetaspaceSize 与 MaxMetaspaceSize

JDK 8 之后永久代被 元空间 Metaspace 替代，主要存放类元数据、方法元信息、常量池相关结构等。元空间使用本地内存，不再直接占用 Java Heap。

  • -XX:MetaspaceSize：触发元空间 GC 的初始阈值，不是固定初始占用。
  • -XX:MaxMetaspaceSize：元空间最大上限；不设置时理论上受限于本机可用内存。
java -XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m -jar app.jar
面试重点：如果系统大量动态生成类，例如 CGLIB、Javassist、Groovy 脚本、频繁热部署，可能导致元空间增长甚至 OutOfMemoryError: Metaspace。
💻

GC 选择与日志诊断参数

JVM 调优不是靠感觉，必须有 GC 日志和监控数据支撑。常见组合如下：

# JDK 8 常见 GC 日志
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/data/logs/gc.log

# JDK 9+ 统一日志
-Xlog:gc*:file=/data/logs/gc.log:time,uptime,level,tags
  • -XX:+UseG1GC：启用 G1，适合较大堆、希望可预测停顿的服务。
  • -XX:MaxGCPauseMillis=200：设置期望最大暂停时间，注意这是目标不是绝对承诺。
  • -XX:+UseParallelGC：吞吐优先，适合后台批处理任务。
  • -XX:+UseConcMarkSweepGC：CMS 在老版本常见，但 JDK 9 已废弃、JDK 14 移除。
调优闭环：观察 GC 日志 → 判断 Minor/Full GC 频率和耗时 → 分析对象生命周期 → 调整堆、年轻代、GC 器或代码分配模式。
⚠️

OOM 诊断参数：线上必备

生产环境建议提前配置 OOM 现场保留参数，否则出问题后只能靠日志猜。

-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/dump/heap.hprof
-XX:ErrorFile=/data/logs/hs_err_pid%p.log
-XX:+ExitOnOutOfMemoryError
  • HeapDumpOnOutOfMemoryError：OOM 时自动导出堆快照，便于 MAT/VisualVM 分析。
  • HeapDumpPath：指定 dump 路径，确保磁盘空间充足且应用有写权限。
  • ErrorFile：JVM 崩溃时输出错误日志。
  • ExitOnOutOfMemoryError：OOM 后直接退出，让容器/守护进程拉起，避免服务处于半死不活状态。
踩坑：Heap Dump 文件可能非常大，2G 堆的 dump 可能接近数 GB；磁盘打满会引发二次故障。
⚖️

容器环境下的 JVM 参数注意点

在 Docker/K8s 中，JVM 看到的不一定是宿主机资源，而是 cgroup 限制。现代 JDK 已增强容器感知，但仍建议显式规划内存。

  • 堆内存不是进程全部内存，还要给 元空间、线程栈、DirectBuffer、JIT CodeCache、native 内存 预留空间。
  • 容器内存 2G 时，不建议直接 -Xmx2g，否则容易被 OOM Killer 杀掉。
  • 可使用 -XX:MaxRAMPercentage、-XX:InitialRAMPercentage 让 JVM 按容器内存比例设置堆。
-XX:InitialRAMPercentage=50.0
-XX:MaxRAMPercentage=70.0

经验值：对于 Web 服务，堆通常控制在容器限制的 50%～75%，剩余空间留给堆外和系统开销。

面试真题：线上服务 JVM 参数怎么配置？

参考回答结构：

  1. 先看服务类型：Web 服务、批处理、低延迟服务的目标不同。
  2. 再看运行环境：物理机还是容器，内存限制是多少，是否有多实例部署。
  3. 基础参数：-Xms 与 -Xmx 一般设为相同；元空间设置上限；开启 GC 日志和 OOM dump。
  4. GC 器选择：大多数中大型服务可优先考虑 G1；吞吐型批任务可考虑 Parallel GC。
  5. 最后压测和线上观测：关注 QPS、RT、GC 次数、GC 总耗时、Full GC、内存水位。
加分表达：JVM 参数没有银弹，我会通过压测和 GC 日志验证，而不是直接套模板。
❓

今日速记清单

  • -Xms：初始堆；-Xmx：最大堆；生产常设相等。
  • -Xmn：年轻代大小；G1 下通常不建议强行固定。
  • MetaspaceSize 是触发 GC 的阈值；MaxMetaspaceSize 是元空间上限。
  • 必须开启 GC 日志、OOM dump、错误日志，便于问题复盘。
  • 容器中要给非堆内存留空间，不能把 -Xmx 顶满容器限制。
