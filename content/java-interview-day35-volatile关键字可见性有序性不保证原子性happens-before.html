核心概念

volatile 是Java提供的最轻量级的同步机制，保证可见性和有序性，但不保证原子性。

本质：volatile 修饰的变量，在读取和写入时都会直接操作主内存（Main Memory），不经过线程工作内存的缓存，从而保证多个线程之间对变量的修改是立即可见的。

volatile 适用于一个线程写、多个线程读的场景，但不适用于多个线程同时写的场景（此时需要加锁或使用Atomic类）。

🔍

可见性原理

Java内存模型（JMM）规定：

  • 每个线程有自己的工作内存（本地缓存）
  • 线程对共享变量的所有操作必须在工作内存中进行
  • 不同线程之间无法直接访问对方的工作内存
  • 线程间变量值的传递需要通过主内存完成

当变量被 volatile 修饰时：

  • 写操作：立即将工作内存中的值强制刷新到主内存
  • 读操作：从主内存中重新读取最新值，而不是用工作内存的缓存
底层实现：volatile 的可见性是通过添加 Lock 前缀指令实现的。该指令会立即将缓存行数据写回主内存，并使其他CPU核心中缓存了该变量的缓存行失效（MESI缓存一致性协议）。
⚙️

有序性 & happens-before

happens-before 规则是JMM中判断数据是否存在竞争、线程是否安全的主要依据。

volatile 相关规则：

Volatile变量规则：对一个volatile变量的写操作 happens-before 于后续对同一变量的读操作。

这就保证了：写线程写入volatile变量前的所有操作，对读线程都是可见的。

禁止指令重排序：

编译器/处理器为了优化性能，可能会对指令进行重排序（Reordering）。volatile 通过内存屏障禁止重排序：

// 写volatile变量时，JMM会在写操作前后插入StoreStore和StoreLoad屏障
// 读volatile变量时，JMM会在读操作前后插入LoadLoad和LoadStore屏障

四种内存屏障：

  • StoreStore：禁止前面的普通写与后面的volatile写重排序
  • StoreLoad：禁止前面的volatile写与后面可能的volatile读/写重排序
  • LoadLoad：禁止后面的普通读与前面的volatile读重排序
  • LoadStore：禁止后面的普通写与前面的volatile读重排序
⚠️

不保证原子性

这是 volatile 最常见的面试考点和踩坑点。

关键理解：volatile 不保证复合操作的原子性！即使变量被 volatile 修饰，count++ 这样的操作仍然是线程不安全的。

原因分析（count++ 包含三步）：

  1. 读取 count 的值（可见性保证）
  2. 对 count 加1（不是原子操作）
  3. 写回 count 的新值（可见性保证）
举例：两个线程同时执行 count++，初始值0。
线程A和线程B都读取到0，各自加1后写回，结果count=1而不是2。
因为 volatile 不能保证读取-修改-写入这个操作序列是原子的。

解决方案：

  • 使用 synchronized 加锁
  • 使用 AtomicInteger / AtomicLong 等原子类（基于CAS实现）
  • 使用 ReentrantLock 显示锁
⚖️

经典使用场景

1. 状态标志位（最常用）

volatile boolean running = true;

// 线程1
while (running) {
    // 执行任务
}

// 线程2（控制线程）
void stop() { running = false; }

2. 双重检查锁定（DCL）——单例模式

class Singleton {
    private static volatile Singleton instance;
    
    public static Singleton getInstance() {
        if (instance == null) {
            synchronized (Singleton.class) {
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
为什么需要volatile？
instance = new Singleton() 并不是一个原子操作，JVM会做3件事：分配内存、初始化对象、将引用指向内存。指令重排序可能导致初始化和赋值互换，另一个线程会拿到一个未初始化完成的对象！volatile禁止了这个重排序。

3. 独立观察变量（发布/订阅模式）

volatile int lastValue;

// 监听线程可以实时看到最新变化

面试高频真题

Q1：volatile 能保证原子性吗？举例说明。
不能。比如 count++ 不是原子操作，需要 synchronized 或 AtomicInteger。
Q2：volatile 和 synchronized 的区别？
volatile 只保证可见性和有序性，不阻塞线程；synchronized 保证可见性、有序性和原子性，会阻塞线程。
Q3：volatile 是如何保证可见性的？
底层通过Lock前缀指令 + MESI缓存一致性协议实现——写操作强制刷回主存，读操作禁用缓存直接从主存读取。
Q4：单例模式的 DCL 为什么要用 volatile？
防止指令重排序导致拿到未初始化的对象。new操作不是原子的，重排序后可能先赋值引用再初始化对象。
Q5：volatile 如何防止指令重排序？
通过插入内存屏障（Memory Barrier），在volatile写前后插入StoreStore/StoreLoad屏障，在volatile读前后插入LoadLoad/LoadStore屏障。
⚠️

实际踩坑经验

踩坑1：用volatile修饰引用类型，认为引用的内部状态也安全。
volatile只保证引用本身的可见性，不保证引用对象内部字段的可见性。如果list本身是volatile的，对list.add()的操作仍然不安全。
踩坑2：认为volatile性能开销很小，大量使用在热点路径上。
volatile变量写操作比普通变量慢约20-100倍（要强制刷缓存、使其他核心缓存失效），虽然比锁快，但也不是免费的。
踩坑3：用volatile解决所有并发问题。
volatile适用范围非常有限。最常见的错误是用volatile做计数器、累加器的线程安全保证。记住：一写多读用volatile，多写多读用锁或Atomic。
