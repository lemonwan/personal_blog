08 GC Roots 概念

📖核心概念：什么是 GC Roots？

在 JVM 的可达性分析算法中，GC Roots 是垃圾回收的起点对象集合。从这些对象出发，通过引用链向下遍历，能够到达的对象标记为存活，不可达的对象则为垃圾。

核心思想

不是孤立地判断对象是否被引用，而是从一组固定的"根"对象出发，看能否找到通往目标对象的路径。

🔒可作为 GC Roots 的对象类型

以下几种对象可以作为 GC Roots：

  • 栈帧中的局部变量表 — 正在执行的方法中的局部变量引用的对象
  • 静态变量引用的对象 — 方法区中类的静态属性引用的对象
  • 常量引用的对象 — 方法区中常量池里的引用（如 String 常量池中的字符串）
  • JNI 引用 — Native 方法栈中 JNI 引用的对象
  • 活跃线程对象 — 当前所有活跃的 Thread 对象
  • 被 synchronized 持有的对象 — 作为 monitor 锁的对象
  • JVM 内部引用 — 系统类加载器、基本类型的 Class 对象、常驻异常对象等

🔬可达性分析算法流程

可达性分析从 GC Roots 出发，采用 BFS/DFS 遍历对象图，标记所有可达对象。未标记的对象即为不可达的垃圾对象。

重要：Stop-The-World

可达性分析必须在 安全点（SafePoint） 执行，此时所有用户线程暂停（STW），才能保证对象引用关系在分析过程中不变。这也是 GC 停顿的主要原因之一。

⚖️引用计数 vs 可达性分析（回顾对比）

对比维度引用计数法可达性分析
原理每个对象维护引用计数器，为0时回收从 GC Roots 遍历对象图，不可达则回收
循环引用无法解决天然解决
效率高，增量式，无需STW需要STW，有停顿
实现复杂度低高（OopMap、安全点）
代表语言Python、SwiftJava、Go、C#
面试重点

两个对象互相引用会被GC回收吗？ 会！可达性分析从 GC Roots 出发，找不到路径就判定为垃圾，循环引用不影响判断。

🎤面试高频问题

  1. GC Roots 具体包含哪些？ — 7类：栈帧局部变量、静态变量、常量引用、JNI引用、活跃线程、synchronized持有对象、JVM内部引用
  2. 为什么选这些对象作为根？ — 它们是程序执行所必需"活的"对象
  3. 什么是不可达对象？ — 从所有GC Roots出发都无法到达的对象
  4. finalize() 复活机制？ — 对象第一次被标记不可达时，若覆盖了finalize()且未执行，进入F-Queue，执行中可重新关联GC Roots"复活"（仅一次）
  5. 三色标记算法？ — CMS/G1并发标记用：白色（未访问）、灰色（已访问未扫完引用）、黑色（全部完成）
记忆口诀

"栈静常，JNI线" — 栈帧局部变量、静态变量、常量池引用、JNI引用、活跃线程
