核心概念：什么是 Java 内存泄漏

Java 内存泄漏不是指对象无法分配内存，而是指：某些对象已经不再被业务逻辑需要，但仍然被 GC Roots 直接或间接引用，导致 GC 无法回收，最终表现为老年代持续上涨、频繁 Full GC，甚至 OutOfMemoryError: Java heap space。

  • 内存溢出 OOM：结果，堆空间真的不够用了。
  • 内存泄漏 Leak：原因之一，无用对象被错误引用，长期无法释放。
  • 内存占用高：不一定泄漏，也可能是缓存、批处理、流量高峰或参数设置不合理。

面试中要强调：Java 有 GC，但 GC 只能回收不可达对象，不能替程序员判断“这个可达对象是否还有业务价值”。

⚙️

排查主线：先看趋势，再抓现场，最后定位引用链

  1. 确认现象：通过监控或 jstat -gcutil 观察堆、老年代、Full GC 次数和耗时是否持续增长。
  2. 确认是否可回收：手动触发或等待 Full GC 后，若老年代占用仍回不到低水位，并呈阶梯式上涨，泄漏概率较高。
  3. 保留现场：导出 Heap Dump，必要时同时保留 GC 日志、线程栈、应用日志和 JVM 参数。
  4. 离线分析：使用 MAT、VisualVM、JProfiler 等工具查看大对象、重复对象、支配树和 GC Roots 引用链。
  5. 回到代码：根据引用链定位集合、缓存、ThreadLocal、监听器、连接资源等持有点，修复并压测验证。

一句话总结：监控判断趋势，Dump 保留证据，MAT 找支配对象，引用链定位代码。

💻

常用命令：如何导出 Heap Dump

线上排查时，常用以下方式导出堆快照：

# 查看 Java 进程
jps -l

# 导出 live 对象，通常更适合分析泄漏
jmap -dump:live,format=b,file=/tmp/app-live.hprof <pid>

# 导出完整堆对象，信息更全但文件更大
jmap -dump:format=b,file=/tmp/app-full.hprof <pid>

# 使用 jcmd，生产环境更推荐
jcmd <pid> GC.heap_dump /tmp/app.hprof

建议 JVM 增加兜底参数：

-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/dumps
-XX:+ExitOnOutOfMemoryError

注意：Dump 可能触发 Stop-The-World，并生成数 GB 文件，生产环境要评估磁盘空间和业务低峰窗口。

MAT 分析重点：Histogram、Dominator Tree、Leak Suspects

  • Histogram：按类统计对象数量和浅堆大小，适合发现异常多的对象类型，如大量 HashMap$Node、byte[]、业务 DTO。
  • Dominator Tree：查看谁“支配”了大量内存。重点看 Retained Heap，它表示释放该对象后可一并释放的内存。
  • Leak Suspects Report：MAT 自动生成可疑泄漏报告，适合作为入口，但不能完全依赖。
  • Path to GC Roots：定位对象为什么没有被回收。通常选择排除弱引用、软引用后的强引用链。

面试回答要区分两个指标：Shallow Heap 是对象自身占用；Retained Heap 是对象及其支配对象释放后可回收的总内存，排查泄漏更关注 Retained Heap。

VisualVM 怎么用：快速观察与现场辅助

VisualVM适合本地、测试环境或可接受 GUI 连接的场景：

  • 通过 Monitor 观察堆、Metaspace、线程数量和 GC 活动。
  • 通过 Sampler/Profiler 查看对象分配热点和 CPU 热点。
  • 生成 Heap Dump 后查看类实例数量、对象引用关系。
  • 结合线程页签观察是否存在大量线程、死锁或线程池堆积。

生产环境更常见做法是：服务器上用 jcmd/jmap 导出 dump，再下载到本地用 MAT/VisualVM 分析，避免远程 GUI 对线上服务造成额外风险。

⚖️

高频泄漏场景：面试要能举例

  • 静态集合：static Map/List 只放不删，生命周期等同 ClassLoader。
  • 本地缓存：没有容量上限、过期策略或淘汰策略，热点变化后旧数据长期驻留。
  • ThreadLocal：线程池复用线程，业务对象放入后未 remove()，被 ThreadLocalMap 长期持有。
  • 监听器/回调：注册后未反注册，事件源持续引用业务对象。
  • 连接与流资源：数据库连接、文件流、Socket 未关闭，可能同时导致堆外资源泄漏。
  • 类加载器泄漏：热部署、插件化场景中，老 ClassLoader 被静态变量、线程、JDBC Driver 等引用。

经典面试题：线上 OOM 你怎么排查？

回答框架：

  1. 先保护现场：确认是否开启 HeapDumpOnOutOfMemoryError，保留 dump、GC 日志、应用日志、JVM 参数。
  2. 看监控：堆内存、老年代、Full GC、QPS、接口耗时、线程数是否异常。
  3. 判断类型：是 Java heap space、Metaspace、Direct buffer memory 还是 unable to create new native thread。
  4. 分析 dump：MAT 查看 Dominator Tree、Retained Heap、Path to GC Roots，定位谁持有大量对象。
  5. 结合代码修复：清理集合、限制缓存、ThreadLocal remove、关闭资源、修复异步队列堆积等。
  6. 验证：压测复现，观察 Full GC 后低水位是否稳定，发布后继续监控。

加分点：说明 Dump 会影响服务，生产要在低峰或摘流后操作；大堆建议使用 MAT 的索引和离线分析机器。

⚠️

踩坑经验：别把所有内存问题都叫堆泄漏

  • 堆外内存：Netty、NIO DirectByteBuffer、Unsafe 分配可能导致 Direct buffer memory，Heap Dump 不一定能完整反映。
  • Metaspace OOM：通常与动态生成类、类加载器泄漏、热部署有关，不是普通 Java 对象泄漏。
  • 线程过多：每个线程都有栈内存，线程池无界创建会导致 native memory 不足。
  • 瞬时大对象：例如大文件一次性读入、超大 SQL 查询结果集，可能是峰值问题而不是长期泄漏。
  • 只看对象数量不够：数量多不一定占内存大，排查时优先关注 Retained Heap 和引用链。
❓

实战建议：如何预防内存泄漏

  • 本地缓存使用 Caffeine/Guava，并设置最大容量、过期时间和监控指标。
  • 线程池中的 ThreadLocal 必须在 finally 中 remove()。
  • 集合、队列、Map 要有上限，避免无界堆积。
  • 资源使用 try-with-resources，确保连接、流、游标关闭。
  • 上线前压测长时间运行，观察 Full GC 后内存低水位是否持续抬升。
  • 生产开启 GC 日志、OOM 自动 dump，并配置磁盘清理与告警。

记忆口诀：缓存有限、ThreadLocal 清、资源必关、队列有界、Dump 留证、引用链定位。
