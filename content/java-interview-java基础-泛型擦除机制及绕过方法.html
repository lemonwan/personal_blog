16 泛型擦除机制及绕过方法

📖泛型擦除机制

  • 什么是泛型擦除？ Java 泛型只在编译期有效，编译后泛型信息会被擦除为原生类型（raw type），例如 List<String> 编译后变为 List。
  • 擦除规则：
    • 无界泛型擦除为 Object
    • 有界泛型擦除为指定的上界类型
    • 类型变量被其第一个边界类型替换
  • 为什么设计为擦除？ 为了向后兼容 JDK 1.5 之前的代码，同时保持 JVM 的简单性。
示例
public class Box<T> { private T value; public T getValue() { return value; } public void setValue(T v) { value = v; } } // 编译后等效于 class Box { private Object value; public Object getValue() { return value; } }

🔬泛型擦除的绕过方法

  • 方法一：通过反射获取泛型信息
    • 反射可以获取 ParameterizedType 中的类型参数
    • 适用于父类/接口的泛型参数提取
    • Spring 的 HibernateTemplate 正是利用此原理
  • 方法二：传入 Class<T> 标记
    public class TypeReference<T> { final Type type; protected TypeReference() { this.type = ((ParameterizedType) getClass() .getGenericSuperclass()).getActualTypeArguments()[0]; } }
  • 方法三：Kotlin 的 reified 类型 Kotlin 通过内联函数实现真正的 reified 泛型。
面试高频

问：new ArrayList<String>() 运行时是 ArrayList<String> 还是 ArrayList？
答：运行时擦除为 ArrayList，类型安全由编译器在编译期保证。

🔒反射的性能开销与框架应用

  • 反射的性能开销：
    • 反射比直接调用慢 10~100倍
    • 主要开销来自 Method.invoke() 的运行时检查、getDeclaredMethod() 的缓存查找
    • 频繁反射会导致 GC 压力增大（产生大量临时对象）
  • 性能优化手段：
    • 使用 method.setAccessible(true) 跳过访问检查
    • 缓存 Method/Field 对象，避免重复查找
    • JDK 9+ 的 MethodHandles 和 VarHandle 提供了更快的替代方案
Spring 中的反射应用
  • IOC 容器：通过反射实例化 Bean
  • DI 注入：反射设置字段
  • AOP：动态代理底层使用反射调用目标方法
  • @Autowired：扫描字段并反射注入依赖

🎤注解与元注解实战

  • 注解（Annotation） 是代码中的元数据，运行时可以被反射获取。
  • 元注解：
    元注解作用
    @Target限定注解可以使用的地方（FIELD, METHOD, TYPE等）
    @Retention保留策略：SOURCE/CLASS/RUNTIME
    @Documented生成 Javadoc 时包含此注解
    @Inherited注解可以被子类继承
    @Repeatable允许同一位置声明多次
面试必问：@Transactional 为什么必须用 RUNTIME？

因为事务管理器需要在运行时通过反射读取 @Transactional 注解，动态创建 AOP 代理并控制事务边界。

自定义注解示例
@Target(ElementType.METHOD) @Retention(RetentionPolicy.RUNTIME) @Documented public @interface RateLimit { int value() default 100; String key() default ""; }
