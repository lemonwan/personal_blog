17 try-with-resources 原理

📖try-with-resources 原理

try-with-resources 是 Java 7 引入的语法糖，用于自动关闭实现 AutoCloseable 或 Closeable 接口的资源，避免资源泄漏。

核心原理

  • 编译器在编译期间将 try-with-resources 翻译成 try-finally 块
  • 每个资源在 try 括号中声明，会自动调用 close() 方法
  • 关闭顺序与声明顺序相反（后进先出）
  • 所有 close() 抛出的异常会被 suppressed（压入异常链），最终只抛出 try 块中的主异常
// 旧写法（Java 7 之前） try { BufferedReader br = new BufferedReader(new FileReader("file.txt")); return br.readLine(); } finally { if (br != null) { br.close(); // 可能抛 IOException，丢失原始异常 } } // 新写法（try-with-resources） try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) { return br.readLine(); } // 自动调用 br.close()

面试要点

  • 多资源关闭：多个资源在同一括号内用分号分隔，关闭时逆序关闭
  • Suppressed Exceptions：close() 的异常被抑制，通过 Throwable.getSuppressed() 获取
  • 自定义 Closeable：任何实现 AutoCloseable 接口的类都可被管理
  • finally 仍可用：try-with-resources 中仍可以写 finally 块做额外清理
💡 面试追问

问：如果 try 块和 close() 都抛了异常，最终抛出哪个？
答：抛出 try 块中的主异常，close() 的异常被抑制。Java 9 后可以使用 effective final 变量，不必在 try 括号内重新声明。

// Java 9 之后，引用已是 effectively final，可直接使用 var br = new BufferedReader(new FileReader("file.txt")); try (br) { // 直接使用 br，无需重新声明 return br.readLine(); }

🛠️异常处理最佳实践

核心原则

  • 不吞异常：禁止空的 catch 块，至少记录日志
  • 不要捕获 Exception 或 Throwable：范围太广，应捕获具体的受检/非受检异常
  • 优先使用 try-with-resources：自动管理资源，避免 finally 中 close() 的坑
  • 异常不要作为控制流：异常性能开销大，不要用 try-catch 做条件判断
  • 自定义异常：使用继承 RuntimeException 的自定义异常传达业务语义
  • 异常信息完整：保留原始异常 cause，不要丢弃原始堆栈
反模式 ❌正确做法 ✅
空的 catch 块记录日志或重新抛出
catch (Exception e) 吞掉所有异常catch 具体异常类型
finally 中 returnfinally 只负责清理
异常作为流程控制用条件判断处理正常分支
💡 面试追问

问：finally 块中如果也有异常会怎样？
答：finally 中的异常会覆盖 try/catch 中的异常，原始异常丢失。所以 finally 中也不应抛异常或 return。
