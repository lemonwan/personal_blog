HashMap 底层实现（数组+链表+红黑树，1.8变化、扩容机制、线程安全）
01

核心概念

HashMap 是 Java 中最常用的键值对集合类，基于 哈希表 实现，允许 null 键和 null 值（注意：Key 只能有一个 null，Value 可以有多个 null）。

底层结构演变：

  • JDK 1.7：数组 + 链表（链表头插法）
  • JDK 1.8+：数组 + 链表 + 红黑树（链表尾插法）

关键参数：

  • 默认初始容量：16（必须是 2 的幂）
  • 负载因子（loadFactor）：0.75（容量与负载因子的乘积 = threshold，达到时触发扩容）
  • 树化阈值：链表长度 ≥ 8 时转换为红黑树
  • 退链阈值：红黑树节点 ≤ 6 时退化为链表
💡 为什么容量必须是 2 的幂？
便于用位运算 (n - 1) & hash 替代取模运算 hash % n，大幅提升性能。同时能保证 hash 值均匀分布，减少哈希冲突。
02

底层原理深度剖析

1. put 流程（JDK 1.8）：

// put(K key, V value) 核心逻辑伪码 if (key == null) return putVal(hash(null), value); hash = hash(key); index = (table.length - 1) & hash; // 计算桶索引 // 遍历桶中的链表/树 if (bucket 为空) {{ 直接插入新节点 }} else if (key 命中) {{ 覆盖 value }} else if (是 TreeNode) {{ 红黑树插入 }} else {{ 链表尾部插入，长度≥8 尝试树化 }} if (size >= threshold) resize(); // 扩容

2. hash() 方法（扰动函数）：

static final int hash(Object key) {{ int h; return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16); }} // 高16位与低16位异或，让高位也参与运算 // 解决低位数不足时高位信息丢失的问题

3. 扩容机制（resize）：

  • 新容量 = 旧容量 × 2（始终为 2 的幂）
  • JDK 1.8 优化：不需要重新计算 hash，只需判断 (oldCap & oldHash) == 0
  • 节点要么在原位置，要么在原位置 + 旧容量
  • 扩容是耗时操作，预估数据量可提前设置初始容量
// 扩容后的位置计算（无需重新hash！） if ((e.hash & oldCap) == 0) {{ // 在原索引位置 newTab[e.hash] = e; }} else {{ // 在新索引位置（原索引 + oldCap） newTab[e.hash + oldCap] = e; }}

4. 为什么 JDK 1.8 改用尾插法？

⚠️ 头插法在多线程下会导致环形链表，造成死循环！
JDK 1.7 的头插法在多线程扩容时可能反转链表顺序，产生环。JDK 1.8 改为尾插法后避免了此问题。但 HashMap 仍不是线程安全的！

HashMap vs LinkedHashMap vs TreeMap：

  • HashMap — 无序，插入/查询 O(1)，最常用
  • LinkedHashMap — 按插入顺序（或访问顺序）遍历，内部维护双向链表
  • TreeMap — 按 Key 自然顺序/自定义 Comparator 排序，基于红黑树，O(log n)
05

实战场景与踩坑经验

🔧 常见应用场景
  • 数据缓存：本地缓存、LRU 缓存（结合 LinkedHashMap 的 accessOrder）
  • 统计计数：词频统计、用户行为计数
  • 去重：利用 key 的唯一性
  • 配置映射：将配置键值对映射为对象属性
  • 缓存中间结果：避免重复计算（Memoization）
💥 踩坑经验
坑1 — 自定义 Key 忘记重写 hashCode/equals：
使用自定义对象作为 Key 时，必须同时重写 hashCode() 和 equals()，否则可能导致数据找不到或重复存储。
坑2 — 并发修改导致 ConcurrentModificationException：
迭代器遍历时调用 map.remove() 而非 iterator.remove()，会触发 fail-fast 机制。
坑3 — 大 HashMap 序列化/传输开销：
超大 HashMap 做 RPC 传输时，序列化时间可能远大于业务处理时间。考虑分页或压缩。
06

模拟面试题

🎯 面试实战
Q1：请手写一个 HashMap，要求支持 put/get/resize 操作。
考察点：对底层数组+链表结构的理解、hash 计算方法、扩容逻辑。
Q2：HashMap 的 hash 方法为什么要将高16位和低16位异或？如果只用低16位会怎样？
考察点：扰动函数设计、数据分布均匀性。答案：如果容量为16，只用低4位会导致高16位信息丢失，分布不均匀，冲突率上升。
Q3：为什么自定义对象作为 HashMap key 时推荐使用 immutable 对象？
考察点：hashCode 缓存、不可变性保证一致性。一旦对象放入 Map，如果其 hashCode 改变了，将永远找不到该 entry。
📝 面试技巧：回答 HashMap 问题时，按照「结构 → 插入流程 → hash 计算 → 扩容 → 线程安全」的框架展开，展现系统化的知识体系。
