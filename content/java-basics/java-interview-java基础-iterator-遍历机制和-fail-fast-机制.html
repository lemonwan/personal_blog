12 Iterator 遍历机制和 fail-fast 机制

📖Iterator 基本用法与遍历机制

Iterator（迭代器）是 Java 集合框架的核心接口，提供了一种统一的遍历集合元素的方式，屏蔽了底层数据结构差异。

基本用法：

for (String item : list) { System.out.println(item); } // 等价于底层代码： Iterator<String> it = list.iterator(); while (it.hasNext()) { String item = it.next(); System.out.println(item); }

核心 API：

  • hasNext() — 判断是否还有下一个元素
  • next() — 返回当前元素并将指针后移
  • remove() — 删除 lastReturned 元素（调用 next 后才能调用）
💡 增强 for 循环的真相
增强 for 循环（foreach）在集合遍历时会被编译器自动转换为 Iterator 遍历；在数组遍历时会转换为索引循环。

🔬fail-fast 机制详解

fail-fast（快速失败）是 Java 集合框架的一种安全机制，用于检测集合在遍历过程中被并发修改的情况。

核心原理：

  • 每个可遍历的集合类内部维护一个 modCount 字段，记录结构修改次数（add/remove/size 变化时递增）
  • Iterator 创建时会保存当前的 expectedModCount = modCount
  • 每次调用 next() 时，检查 expectedModCount == modCount
  • 如果不相等，立即抛出 ConcurrentModificationException
(// Iterator.java 伪代码) public E next() { // 检查是否被并发修改 checkForComodification(); // ... 返回元素 } final void checkForComodification() { if (modCount != expectedModCount) throw new ConcurrentModificationException(); }

结构修改 vs 非结构修改：

  • 结构修改：add、remove、clear 等改变集合大小的操作 → 触发 modCount 变更 → fail-fast
  • 非结构修改：set、replace 等操作不改变集合大小 → 不触发 fail-fast

🔒fail-safe 机制

与 fail-fast 相对的是 fail-safe（安全失败），其核心思想是对集合的快照（copy）进行遍历，遍历过程中修改原集合不会影响遍历结果。

主要使用场景：

  • ConcurrentHashMap — 遍历整个 table 的 snapshot
  • CopyOnWriteArrayList — 每次修改都创建新数组的副本（写时复制）
  • CopyOnWriteArraySet — 基于 CopyOnWriteArrayList

🛠️遍历时安全的删除操作

在遍历过程中删除元素，有三种安全方式：

方法一：使用 Iterator.remove()（推荐）

Iterator<String> it = list.iterator(); while (it.hasNext()) { String s = it.next(); if (needRemove(s)) { it.remove(); // 安全！内部会同步 modCount } }

方法二：使用 Java 8+ removeIf()

list.removeIf(s -> s.startsWith("prefix"));

方法三：正向索引遍历，倒序删除

for (int i = list.size() - 1; i >= 0; i--) { if (needRemove(list.get(i))) { list.remove(i); } }
⚠️ 常见错误：不要在 foreach 中直接 remove
以下写法会导致 ConcurrentModificationException：
list.stream().forEach(item -> { if (condition) { list.remove(item); // ❌ 错误！直接操作原集合 } });

🎤面试高频考点

Q1：Iterator 和 ListIterator 有什么区别？

  • Iterator 只能单向遍历（next），ListIterator 可双向遍历（next/previous）
  • ListIterator 支持 add、set、remove 操作
  • ListIterator 只适用于 List 集合

Q2：为什么 foreach 不能安全删除元素？

  • 编译器会将 foreach 转换为 Iterator 遍历
  • 在循环体内直接调用集合的 remove() 会改变 modCount
  • 下次调用 next() 时发现 modCount != expectedModCount，抛出异常

Q3：fail-fast 和线程安全的关系？

  • fail-fast 不是线程安全机制，它是单线程检测并发修改的手段
  • 即使多线程环境下不抛 ConcurrentModificationException，也不代表线程安全
  • 真正的线程安全需要使用 ConcurrentHashMap、CopyOnWriteArrayList 等并发类

Q4：Spliterator 和 Iterator 的区别？

  • Spliterator 是 Java 8 引入的"可分割迭代器"，支持并行遍历
  • Spliterator 可以拆分（trySplit），适合 Stream 并行处理
  • Spliterator 是 fail-fast 的，但更强调并行能力

🔬源码深入：ArrayList 的 Iterator 实现

ArrayList 的 Iterator 实现源码（关键部分）：

// ArrayList.java 内部类 private class Itr implements Iterator<E> { int cursor = 0; // 下一个元素的索引 int lastRet = -1; // 上一个返回元素的索引 int expectedModCount = modCount; // 快照 public E next() { checkForComodification(); int i = cursor; if (i >= size) throw new NoSuchElementException(); Object[] elementData = ArrayList.this.elementData; if (i >= elementData.length) throw new ConcurrentModificationException(); cursor = i + 1; return (E) elementData[lastRet = i]; } public void remove() { if (lastRet < 0) throw new IllegalStateException(); checkForComodification(); try { ArrayList.this.remove(lastRet); cursor = lastRet; lastRet = -1; expectedModCount = modCount; // 关键：同步 modCount } catch (IndexOutOfBoundsException ex) { throw new ConcurrentModificationException(); } } final void checkForComodification() { if (modCount != expectedModCount) throw new ConcurrentModificationException(); } }
🔑 核心要点
Iterator.remove() 调用的是集合自己的 remove 方法后，手动将 expectedModCount = modCount 重新同步，这是唯一安全的遍历中删除方式。
