Day 15 — 接口默认方法/静态方法

模块：Java 基础 · Java 8+ 新特性
15 接口默认方法/静态方法

📖接口默认方法（default methods）

Java 8 引入了 default 关键字，允许在接口中定义带有默认实现的方法。

核心目的：在不破坏已有实现类的前提下，对接口进行演进升级。

  • 语法：default 修饰方法，提供默认实现
  • 实现类：可以继承接口的默认实现，也可以覆写
  • 优先级：类的方法 > 接口的默认方法 > 父接口的 default 方法
💡 经典场景
Java 8 的 Collection 接口大量使用了 default 方法（如 stream()、forEach()、removeIf()），在不破坏已有类（如 ArrayList、HashMap）的前提下新增了函数式能力。

📖接口静态方法（static methods）

Java 8 还允许在接口中定义 static 方法，直接通过接口名调用。

  • 语法：static 修饰方法，不能有默认实现依赖 this
  • 调用方式：InterfaceName.staticMethod()
  • 区别：默认方法可被实现类覆写，static 方法不能覆写
💡 设计意图
静态方法用于提供与接口相关的工具方法，如 Comparator 中的 comparing()、comparingInt()、reversed() 等工厂方法。

🔒默认方法与继承的冲突解决

当一个类实现多个接口，且这些接口有同名 default 方法时，编译器会报错，需要显式选择。

  • 规则 1：类的方法优先级高于任何接口的 default 方法
  • 规则 2：如果继承的父类提供了实现，优先使用父类的
  • 规则 3：多个接口有同名 default 方法 → 必须显式选择
// 必须通过 super 显式选择 class MyClass implements InterfaceA, InterfaceB { public void doSomething() { InterfaceA.super.doSomething(); // 显式指定 } }
⚠️ 面试考点
这个冲突解决规则体现了 Java 的设计哲学：类的实现优先于接口的契约，同时要求开发者显式处理歧义。

📖函数式接口与 @FunctionalInterface

Java 8 定义了四大核心函数式接口，是 Lambda 表达式的基础。

接口方法签名用途
Supplier<T>T get()供给型：无入参，有出参
Consumer<T>void accept(T t)消费型：有入参，无出参
Function<T,R>R apply(T t)函数型：有入参，有出参
Predicate<T>boolean test(T t)断言型：有入参，返回布尔值
🎤 面试要点
@FunctionalInterface 注解是可选的，但强烈建议加上。它会在编译期检查是否确实只有一个抽象方法，防止接口误用。如果接口中有多个抽象方法，标注了 @FunctionalInterface 会编译报错。

⚖️默认方法 vs 抽象类 vs 工具类

默认方法的出现引发了一个常见设计问题：什么时候用接口 default，什么时候用抽象类或工具类？

维度接口 default抽象类工具类（static）
多实现✅ 支持❌ 不支持✅ 支持
状态维护❌ 不能✅ 可以❌ 不能
向后兼容✅ 完美⚠️ 可能破坏✅ 完美
覆写✅ 可以✅ 可以❌ 不能
💡 最佳实践
接口 default 适合不依赖状态的行为扩展；有状态的行为应使用抽象类；纯工具方法应使用 static 工具类。
