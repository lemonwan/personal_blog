1.1.1 HashMap 底层实现（数组+链表+红黑树，1.8变化）

HashMap 是 Java 中最常用的数据结构之一，面试中几乎必考。理解其底层实现是掌握集合框架的基石。

核心数据结构演变

// JDK 1.7: 数组 + 链表（头插法） +-------+ +----+----+----+ | hash | → |[0]→[1]→[2] | ← 每个位置挂一条链表 +-------+ +----+----+----+ // JDK 1.8+: 数组 + 链表 + 红黑树（尾插法） +-------+ +------------------+ | hash | → | Node | Node ... | ← 链表长度 >= 8 且数组长度 >= 64 +-------+ | ↓ | | TreeNode | ← 转为红黑树 | ↓ | | TreeNode | +------------------+ // 节点结构（Node 实现 Map.Entry） static class Node<K,V> implements Map.Entry<K,V> { final int hash; final K key; V value; Node<K,V> next; }

1.7 → 1.8 关键变化

特性JDK 1.7JDK 1.8+
数据结构数组 + 链表数组 + 链表 + 红黑树
插入方式头插法尾插法
扩容后链表顺序倒序保持原序
树化条件不支持链表长度 ≥ 8 且数组长度 ≥ 64
退化条件不支持红黑树节点 ≤ 6 时退化为链表
线程安全多线程扩容可能死循环尾插法避免死循环，但仍非线程安全
💡 面试要点：1.7 头插法在多线程扩容时可能导致环形链表（死循环），1.8 改为尾插法解决了这个问题。但 HashMap 仍然不是线程安全的！

哈希计算（hash 方法）

// 高 16 位异或低 16 位，减少哈希冲突 static final int hash(Object key) { int h; return (key == null) ? 0 : (h ^ (h >>> 16)); } // 等价于：(h = key.hashCode()) ^ (h >>> 16) // 为什么这样做？ // 数组长度总是 2 的幂，index = (n - 1) & hash // 只取低位参与计算的话，高位信息丢失容易冲突 // 高低位异或后，既保留高位信息又利用低位，分布更均匀

Put 操作完整流程

put(key, value) 流程： ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ 1. 判断数组是否为空，为空则初始化（默认容量 16，阈值 12） 2. 计算 key 的 hash 值 → index = (n - 1) & hash 3. 如果 bucket[i] 为空，直接新建 Node 放入 4. 如果不为空： a) key 与头节点相同 → 直接覆盖 value b) 如果是 TreeNode → 调用 putTreeVal 红黑树插入 c) 如果是链表 → 尾插法遍历插入 - 遍历过程中发现 key 已存在 → 覆盖 value - 插入后链表长度 >= 8 → 检查是否树化 5. 元素个数 > 阈值 (threshold) → 扩容（翻倍） ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
⚠️ 踩坑经验： ① 自定义对象作为 Key 必须重写 hashCode() 和 equals()； ② Key 为 null 时，始终存放在 index=0 的位置； ③ HashMap 允许一个 null key 和多个 null value。

// 扩容时 1.8 的优化：不需要重新 hash！ // 扩容后 index 只有两种可能： // 原 index 或 原 index + oldCapacity // 判断方式：(e.hash & oldCapacity) == 0 ? 原位置 : 原位置+oldCapacity // 链表中的元素只需遍历一次，比 1.7 需要遍历两次更高效

面试模拟

Q1：为什么 HashMap 的容量必须是 2 的幂？
A：① 使得 (n-1) & hash 等价于 hash % n，位运算更高效；② 2 的幂减 1 全为 1，能最大程度利用 hash 值的所有位，减少碰撞。
Q2：为什么负载因子默认是 0.75？
A：0.75 是时间开销和空间开销的折中。值越大，空间利用率高但冲突概率增大；值越小，冲突少但浪费空间。统计表明 0.75 是较优平衡点。
Q3：JDK 1.7 为什么用头插法，1.8 改为尾插法？
A：头插法在多线程扩容时，逆序会导致环形链表产生死循环。尾插法保持原序，避免此问题。同时 1.8 引入红黑树后，头插法会反转树的遍历顺序，尾插法则不会。
Q4：HashMap 扩容的 time complexity？
A：put/get 平均 O(1)，最差（全部碰撞入树）O(log n)。扩容本身 O(n)，均摊后不影响整体效率。
1.1.2 ConcurrentHashMap 分段锁到 CAS+synchronized 演进

三个版本的演进路线

版本线程安全方案粒度特点
JDK 1.7分段锁 (Segment)Segment 级别继承 ReentrantLock，默认 16 个 Segment，最多 16 线程并发
JDK 1.8 (初版)CAS + synchronized (Node 级别)Node 级别放弃 Segment，锁粒度细化到桶节点，并发度大幅提升
JDK 1.8 (树化后)synchronized (TreeNode 级别)TreeNode 级别红黑树节点锁竞争更小
// 1.7 Segment 结构 ┌─────────────┬──────────────────────────┐ │ Segment[0] │ HashMap: Node[] table │ │ (继承锁) │ 每个 Segment 独立锁 │ ├─────────────┼──────────────────────────┤ │ Segment[1] │ ... │ └─────────────┴──────────────────────────┘ // 最大并发度 = Segment 数量（默认 16） // 1.8 Node 结构 — 无 Segment，直接锁桶 static class Node<K,V> { volatile V value; // volatile 保证可见性 final int hash; final K key; volatile Node<K,V> next; // volatile 防扩容时链表断裂 } // put 操作: CAS 插入头节点 → 失败则 synchronized 锁头节点
💡 核心原理：1.8 中 put 操作先尝试 CAS 插入新节点（无锁），失败后用 synchronized 锁住桶头节点再插入。get 操作不需要同步，只读取 volatile 字段，因此是无锁读取。

面试要点

  • 为什么 1.8 放弃分段锁？ Segment 数量固定导致并发度上限，且扩容时整个 Segment 锁住影响大。Node 级别锁更细粒度。
  • size() 方法怎么统计？ 1.8 使用 baseCount + CounterCell[]（类似 LongAdder），通过 volatile + CAS 累加，保证近似精确。
  • 与 HashMap 区别： 线程安全、不允许 null key/value、扩容策略不同、size() 为近似值。
Q：ConcurrentHashMap 和 Hashtable 的区别？
A：Hashtable 用 synchronized 锁整张表，并发性能极差；ConcurrentHashMap 1.8 锁粒度为桶级别，并发度接近 Integer.MAX_VALUE。且 Hashtable 不允许 null，ConcurrentHashMap 不允许 null key/value。
1.1.3 ArrayList vs LinkedList 使用场景
维度ArrayListLinkedList
底层结构动态数组双向链表
随机访问 get(i)O(1) ⭐O(n)
尾部插入 add均摊 O(1) ⭐O(1) ⭐
中间插入/删除O(n)（需移动元素）O(1)（找到位置后）
内存占用较紧凑（仅存元素）较大（每个节点 +2 个引用）
缓存友好性高（连续内存）⭐低（分散内存）
线程安全否否
💡 实战场景： ① 读多写少、按索引访问 → ArrayList； ② 频繁在头部/中间插入删除 → LinkedList（但实际项目中 ArrayList 更常用，因为缓存友好，LinkedList 的实际性能未必更好）； ③ Java 9 起 LinkedList 内部结构有所优化。
⚠️ 面试陷阱：不要迷信"LinkedList 插入删除 O(1)"——找到位置本身就是 O(n)，所以实际使用中 ArrayList 在大多数场景下性能更好。
1.1.4 HashSet / TreeSet 底层原理
  • HashSet：底层就是 HashMap，新元素作为 Key 存入 HashMap，Value 固定为 PRESENT（一个 static final Object）。因此 HashSet 也不允许重复元素。
  • TreeSet：底层是 TreeMap（基于红黑树），元素按自然顺序或 Comparator 排序。支持 first()/last()/subSet() 等操作。
  • LinkedHashSet：在 HashSet 基础上加了双向链表维护插入顺序。
Q：如何保证 HashSet 元素的唯一性？
A：hashCode() 和 equals() 双重校验。先比较 hash 值，hash 相同再比较 equals。两个自定义对象要视为相等，必须同时满足 hashCode 相同且 equals 返回 true。
1.1.5 Iterator 遍历机制和 fail-fast 机制

Iterator 工作原理

// 使用示例 Iterator<String> it = list.iterator(); while (it.hasNext()) { String s = it.next(); if (s.equals("target")) { it.remove(); // 正确：通过 Iterator 删除 } } // 错误做法：遍历中直接操作集合会触发 fail-fast for (String s : list) { list.remove(s); // ❌ ConcurrentModificationException

fail-fast 机制

  • Java 集合框架中的迭代器在遍历过程中检测到集合结构被修改（非通过迭代器的 remove），会抛出 ConcurrentModificationException。
  • 实现原理：每个集合维护一个 modCount 字段，每次结构修改时递增。迭代器创建时记录 expectedModCount，每次 next() 时校验两者是否一致。
  • 注意：fail-fast 是快速失败而非安全保证，不能保证 100% 检测到并发修改，仅用于检测 bug。
  • 用 fail-safe 的集合（如 ConcurrentHashMap、CopyOnWriteArrayList）遍历不会因为并发修改抛异常。
💡 面试高频：fail-fast 和 fail-safe 的区别是集合框架的经典考点。HashMap 的迭代器是 fail-fast，ConcurrentHashMap 的是 fail-safe。
