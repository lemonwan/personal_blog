11 异常处理最佳实践

📖异常处理的核心原则

  • 不要忽略异常 — 空 catch 块是严重的代码气味，至少应该记录日志
  • 尽早失败 — 在方法入口处校验参数，尽早抛出异常，避免进入无效状态
  • 异常处理 != 错误处理 — 异常应该只用于异常情况（不常见、不可预期），不能用于控制正常流程
  • 捕获具体异常 — 优先捕获最具体的异常类型，避免直接捕获 Exception 或 Throwable
  • 异常信息要完整 — 提供有意义的错误信息，包含上下文，便于排查问题

🛠️正确的捕获与转换

  • 不要吞掉异常 — 错误地捕获异常却不采取任何措施是最常见的反模式
  • 不要记录后不处理 — 如果记录完日志后什么都不做，调用方无法感知错误
// ❌ 反模式：捕获后什么都不做 try { process(data); } catch (IOException e) { // 什么也不做 } // ✅ 正确：记录日志后重新抛出或返回错误码 try { process(data); } catch (IOException e) { log.error("处理数据失败: {}", data.getId(), e); throw new BusinessException("数据处理失败", e); } // ✅ 或者转换为受检异常抛出 try { process(data); } catch (IOException e) { throw new DataAccessException("数据访问失败", e); }
📋 异常链（Exception Chaining）
始终使用 new SpecificException("message", originalException) 构造函数保留原始异常，这样堆栈跟踪信息才不会丢失。

⚖️try-with-resources vs finally

  • try-with-resources（推荐） — JDK 7+ 引入，自动关闭实现 AutoCloseable 的资源
  • finally — 传统方式，需要手动关闭资源，代码冗长且容易出错
// ✅ JDK 7+ try-with-resources（推荐） try (FileInputStream fis = new FileInputStream("data.txt"); BufferedReader br = new BufferedReader(new InputStreamReader(fis))) { return br.readLine(); } catch (IOException e) { log.error("读取文件失败", e); throw new BusinessException("文件读取失败", e); } // 自动关闭，无需 finally 块
// ❌ 传统方式（不推荐） BufferedReader br = null; try { br = new BufferedReader(new FileReader("data.txt")); return br.readLine(); } catch (IOException e) { throw new RuntimeException(e); } finally { if (br != null) { try { br.close(); } catch (IOException e) { // 这个异常会掩盖前面的异常！ } } }
⚠️ finally 中的异常会掩盖 try 中的异常！try-with-resources 在 JDK 9+ 中支持使用 effectively final 的局部变量，无需重新声明。

🔬自定义异常的设计

  • 何时创建自定义异常 — 当框架/业务层面的错误无法用标准异常充分表达时
  • 继承体系 — 运行时异常继承 RuntimeException，受检异常继承 Exception
  • 命名规范 — 以 Exception 结尾，清晰表达异常场景（如 UserNotFoundException）
  • 携带上下文 — 构造异常时传入有意义的参数（如用户 ID、订单号）
public class BusinessException extends RuntimeException { private final String errorCode; private final Object context; public BusinessException(String message, Throwable cause) { super(message, cause); this.errorCode = determineErrorCode(message); this.context = extractContext(); } public BusinessException(String message, String errorCode) { super(message); this.errorCode = errorCode; } } // 业务层使用 throw new BusinessException("用户不存在", "USER_NOT_FOUND", userId);
📋 分层异常策略
• DAO 层：受检异常，向 Service 层报告数据库错误
• Service 层：运行时异常，向 Controller 层报告业务错误
• Controller 层：统一异常处理器，转换为 HTTP 响应

📖Spring 统一异常处理

  • @RestControllerAdvice + @ExceptionHandler — Spring 推荐的统一异常处理方式
  • @ControllerAdvice — 全局拦截所有控制器的异常
  • 异常映射 — 不同类型的异常映射到不同的 HTTP 状态码
@RestControllerAdvice public class GlobalExceptionHandler { @ExceptionHandler(UserNotFoundException.class) public ResponseEntity<ApiResponse<?>> handleUserNotFound( UserNotFoundException e) { return ResponseEntity.status(HttpStatus.NOT_FOUND) .body(ApiResponse.error(404, "用户不存在")); } @ExceptionHandler(BusinessException.class) public ResponseEntity<ApiResponse<?>> handleBusinessException( BusinessException e) { return ResponseEntity.badRequest() .body(ApiResponse.error(400, e.getMessage())); } @ExceptionHandler(Exception.class) public ResponseEntity<ApiResponse<?>> handleGeneralException( Exception e) { log.error("未预期的异常", e); return ResponseEntity.status(500) .body(ApiResponse.error(500, "服务器内部错误")); } }
💥 注意：@ExceptionHandler 方法必须是非私有（非-private）的，否则不会生效。

❓面试高频问题

  • Q：受检异常和非受检异常的区别？何时用哪种？
    A：受检异常（Exception）编译时检查，用于可恢复的错误（如 IOException）；非受检异常（RuntimeException）编译时不检查，用于程序错误（如 NullPointerException）。现代实践倾向于减少使用受检异常，优先用非受检异常+日志。
  • Q：finally 一定执行吗？
    A：finally 在 return、break、continue 时都会执行。但在以下情况不会执行：System.exit()、JVM 崩溃、进程被 kill、finally 前进入死循环。
  • Q：try-with-resources 的原理？
    A：编译期展开为 try-finally 块，在 finally 中按声明顺序逆序调用 close()，先声明的后关闭，后声明的先关闭。
  • Q：如何选择合适的异常类型？
    A：参数校验错误用 IllegalArgumentException；业务规则违反用自定义运行时异常；资源问题用受检异常（如 IOException）。
  • Q：throw 和 throws 有什么区别？
    A：throw 用于方法体内抛出一个异常实例；throws 用于方法签名声明可能抛出的异常类型。
