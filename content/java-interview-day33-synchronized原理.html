核心概念与作用

synchronized 是 Java 内置的关键字，用于实现线程同步，保证共享资源的原子性、可见性和有序性。

三种使用方式：

  • 修饰实例方法：锁的是当前实例对象（this）
  • 修饰静态方法：锁的是当前类的 Class 对象
  • 修饰代码块：锁的是括号内配置的对象

synchronized 是 可重入锁：一个线程已经获得某个对象的锁后，可以再次获取同一把锁而不会产生死锁。

⚙️

底层实现：Monitor 对象

synchronized 的底层是通过 Monitor（管程） 对象实现的。每个 Java 对象都关联一个 Monitor 对象。

Monitor 的核心结构：

  • _owner：指向持有锁的线程
  • _EntryList：等待获取锁的线程阻塞队列
  • _WaitSet：调用 wait() 后等待的线程集合
  • _recursions：记录重入次数

字节码层面：同步代码块使用 monitorenter 和 monitorexit 指令，同步方法通过 ACC_SYNCHRONIZED 标志隐式实现。

🔍

Mark Word 与锁标志位

Java 对象头的 Mark Word 区域（64位虚拟机占 8 字节）存储了锁状态信息：

锁状态标志位存储内容
无锁01对象hashCode、分代年龄
偏向锁01偏向线程ID、epoch
轻量级锁00指向栈中锁记录的指针
重量级锁10指向 Monitor 的指针
GC标记11空（不使用）

偏向锁的标志位也是 01，通过是否偏向（1）区分。锁只能升级，不能降级。

🔍

锁升级完整流程

1. 偏向锁（Biased Locking）

当第一个线程访问同步块时，通过 CAS 将 Mark Word 设置为偏向自己。后续同一线程进入无需 CAS，直接检查线程ID即可。

2. 轻量级锁（Lightweight Locking）

有多个线程竞争时，偏向锁撤销，升级为轻量级锁：

  • 线程在栈帧中创建 Lock Record
  • CAS 将 Mark Word 复制到 Lock Record
  • CAS 将 Mark Word 指向 Lock Record
  • 成功则获得锁，失败则自旋重试

3. 重量级锁（Heavyweight Locking）

自旋超过 10 次或等待线程过多时，升级为重量级锁，线程真正阻塞，依赖操作系统内核调度。

锁升级是 JDK 6 引入的优化：无锁 → 偏向锁 → 轻量级锁 → 重量级锁

面试高频考点

Q1: synchronized 和 Lock 的区别？

  • synchronized 是关键字，Lock 是接口
  • synchronized 不可中断，Lock 可中断
  • synchronized 非公平，Lock 可选择公平/非公平
  • Lock 可绑定多个 Condition 条件

Q2: 什么是锁消除、锁粗化？

锁消除：JIT 检测到不可能存在共享数据竞争时，自动消除同步

锁粗化：将多个连续的加锁解锁操作合并为一个大范围的锁

Q3: synchronized 可以保证有序性吗？

可以，持有同一锁的线程，解锁操作 happens-before 随后的加锁操作。

实际使用最佳实践

  • 尽量缩小同步范围：只锁必要的临界区代码
  • 避免锁嵌套：防止死锁的发生
  • 优先使用 synchronized：除非需要 Lock 的高级特性
  • 注意锁对象的选择：避免使用 String 常量或基本类型包装类作为锁
// 推荐：锁代码块而非整个方法
public void safeMethod() {
    // 非临界区代码...
    synchronized (this) {
        // 临界区 - 仅包含必要操作
        count++;
    }
    // 非临界区代码...
}

常见踩坑场景

⚠️ 坑1：锁 String 常量

String a = "hello"; String b = "hello"; 指向同一对象，导致意外的锁竞争

⚠️ 坑2：锁 Integer/Long

缓存池范围内的 Integer（-128~127）是同一对象，超出范围才会 new

⚠️ 坑3：锁 this 与锁 class 混淆

实例锁和类锁是两把不同的锁，互不干扰

⚠️ 坑4：异常时未释放锁

synchronized 在异常时会自动释放锁，Lock 则需要 finally 手动释放
