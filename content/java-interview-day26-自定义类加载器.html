核心概念：什么是自定义类加载器

类加载器负责把 .class 字节码加载到 JVM，并生成对应的 Class 对象。自定义类加载器就是继承 ClassLoader，按照业务需要从特定来源读取字节码，例如本地加密 class、网络、数据库、插件目录或隔离目录。

  • Bootstrap ClassLoader：加载核心类库，如 java.lang.*。
  • Platform ClassLoader：加载平台相关类。
  • Application ClassLoader：加载应用 classpath 下的类。
  • Custom ClassLoader：开发者扩展，用于隔离、热部署、加密保护、插件化等场景。

面试时要强调：自定义类加载器不是为了随意替代双亲委派，而是在特定边界内扩展类加载来源和隔离模型。

⚙️

底层原理：loadClass 与 findClass 的分工

ClassLoader 的经典流程是：

  1. loadClass(name, resolve)：入口方法，默认遵循双亲委派，先检查是否已加载，再委托父加载器，最后调用自己的 findClass。
  2. findClass(name)：真正由子类实现从哪里读字节码。
  3. defineClass(name, bytes, 0, len)：把字节数组转换成 JVM 中的 Class 对象。
  4. resolveClass(clazz)：链接阶段的一部分，解析符号引用。

一般推荐重写 findClass，不要直接重写 loadClass，因为直接改 loadClass 很容易破坏双亲委派，导致核心类冲突、类型不兼容或安全问题。

💻

代码模板：最小可用自定义 ClassLoader

public class FileClassLoader extends ClassLoader {
    private final Path baseDir;

    public FileClassLoader(Path baseDir) {
        this.baseDir = baseDir;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            String fileName = name.replace('.', '/') + ".class";
            byte[] bytes = Files.readAllBytes(baseDir.resolve(fileName));
            return defineClass(name, bytes, 0, bytes.length);
        } catch (IOException e) {
            throw new ClassNotFoundException(name, e);
        }
    }
}

调用方式：

ClassLoader loader = new FileClassLoader(Paths.get("/opt/plugins"));
Class<?> clazz = loader.loadClass("com.demo.PluginImpl");
Object obj = clazz.getDeclaredConstructor().newInstance();

注意：同一个全限定类名，如果由不同类加载器加载，在 JVM 看来是不同的类型。

🔍

典型应用场景

  • Tomcat 类隔离：每个 Web 应用使用独立的 WebAppClassLoader，避免不同应用依赖版本互相污染。
  • 插件化系统：主程序定义 SPI 或公共接口，插件 jar 由独立加载器加载，实现动态扩展。
  • 热部署和热加载：卸载旧 ClassLoader，创建新 ClassLoader 加载新版本类。类本身不能单独卸载，能卸载的是加载它的 ClassLoader 及其所有 Class。
  • 字节码加密：读取加密 class 后先解密，再 defineClass。
  • 框架增强：结合 Java Agent、ASM、ByteBuddy 做运行时增强或诊断。

高频面试题与答题要点

Q1：如何实现一个自定义类加载器？

答：继承 ClassLoader，优先重写 findClass，在其中读取字节码并调用 defineClass。如果需要改变委派逻辑才重写 loadClass，但要非常谨慎。

Q2：为什么两个相同类名的对象不能强转？

答：JVM 判断类唯一性依赖类全限定名加加载它的 ClassLoader。类名相同但加载器不同，就是两个不同类型。

Q3：热部署为什么要替换 ClassLoader？

答：JVM 中 Class 加载后通常不能单独卸载，只有当对应 ClassLoader 不可达、且其加载的 Class 和实例都不可达时，才可能被 GC 回收。因此热部署通常创建新的 ClassLoader 加载新类。

Q4：什么时候会破坏双亲委派？

答：例如 Tomcat 为了 Web 应用优先加载自己的依赖；JDBC SPI 通过线程上下文类加载器让父加载器加载的核心代码反向找到应用侧实现类。

⚠️

踩坑经验：类加载器最容易出问题的地方

  • ClassCastException：同名类被不同 ClassLoader 加载后互相强转失败。公共接口应放到父加载器可见的位置。
  • LinkageError：同一个加载器重复定义同名类，或依赖版本冲突。
  • 内存泄漏：热部署后，线程、ThreadLocal、静态字段、Timer、线程池仍引用旧类或旧 ClassLoader，导致卸载失败。
  • 误重写 loadClass：破坏双亲委派后可能加载到伪造的核心类，或造成 JDK 和框架类冲突。
  • 资源加载混乱：getResource、Thread.currentThread().getContextClassLoader() 与当前类加载器不一致，可能读取到错误版本配置。
❓

一句话总结

自定义类加载器的本质是自定义字节码来源，并控制类隔离边界。面试回答时围绕 loadClass、findClass、defineClass、双亲委派、类型唯一性、Tomcat、插件化和热部署场景展开，基本就能答到中高级深度。
