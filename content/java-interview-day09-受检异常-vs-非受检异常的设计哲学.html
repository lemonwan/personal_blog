核心概念

Java 的异常体系以 Throwable 为根，分为两大分支：

  • Error（错误）：JVM 内部严重问题，如 OutOfMemoryError、StackOverflowError，程序无法处理
  • Exception（异常）：程序可处理的问题，分为 受检异常（Checked Exception） 和 非受检异常（Unchecked Exception）

判定标准很简单：看是否继承自 RuntimeException

⚖️

受检异常 vs 非受检异常

对比维度受检异常（Checked）非受检异常（Unchecked）
继承Exception 的子类（不含 RuntimeException）RuntimeException 的子类
编译检查✅ 必须处理（try-catch 或 throws）❌ 可不处理，编译通过
典型例子IOException、SQLException、ClassNotFoundExceptionNullPointerException、ArrayIndexOutOfBoundsException、IllegalArgumentException
处理义务调用者必须处理或上抛调用者可选择处理或忽略
适用场景可恢复的外部故障（IO失败、网络中断、SQL异常）程序逻辑错误（空指针、数组越界、非法参数）
⚙️

设计哲学

受检异常的初衷：“强制程序员考虑异常情况”，提高代码健壮性。—— Java 设计者 James Gosling

✅ 受检异常的优势

  • 编译器强制处理，防止忽略重要错误场景
  • 异常信息在方法签名中可见（throws），调用者一目了然
  • 适合 IO、网络、数据库等可预见的业务异常

❌ 受检异常的争议

  • 导致大量样板代码（try-catch 泛滥），降低可读性
  • 调用链上每层都要处理或上抛，修改底层异常类型影响面大
  • 在 Lambda/Stream 中处理繁琐

业界趋势：现代框架（Spring、Hibernate）已大量转向非受检异常。例如 Spring 将 SQLException 包装为 DataAccessException（RuntimeException），给上层更多灵活性。

💻

最佳实践

// ✅ 推荐：使用非受检异常，通过统一异常处理机制捕获
public void process() {
    if (invalid) {
        throw new BusinessRuntimeException(业务异常);
    }
}

// 在 Spring 中统一处理
@ControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(BusinessRuntimeException.class)
    public ResponseEntity handle(BusinessRuntimeException e) {
        return ResponseEntity.badRequest().body(e.getMessage());
    }
}

业界推荐原则

  • 推荐 调用方能合理恢复的场景 → 使用受检异常（如重试连接、使用备选方案）
  • 不推荐 调用方无法恢复的逻辑错误 → 使用非受检异常（如空指针、非法参数）
  • 推荐 框架/底层库 → 用非受检异常包装后抛出，由上层统一处理
  • 注意 不要在 finally 中抛异常，会覆盖原始异常（可用 try-with-resources 规避）
⚠️

常见陷阱

陷阱1：在 finally 中返回或抛异常

// 原始异常被吃掉
try {
    reader = new BufferedReader(new FileReader(file.txt));
    return reader.read();
} finally {
    if (reader != null) reader.close(); // 这里抛异常会覆盖 try 中的异常
}

// ✅ 正确：使用 try-with-resources
try (BufferedReader reader = new BufferedReader(new FileReader(file.txt))) {
    return reader.read();
}

陷阱2：捕获异常不做任何处理

// ❌ 静默吞异常
catch (Exception e) {
    // 空的！异常被吞掉，排查困难
}

// ✅ 至少打印日志
catch (Exception e) {
    log.error(操作失败, e);
    throw new BusinessRuntimeException(操作失败, e);
}

面试常见问题

Q1：你们项目中用受检异常还是非受检异常？为什么？

回答要点：目前项目主要用非受检异常。配合 Spring @ControllerAdvice + @ExceptionHandler 统一处理，代码更简洁。对于不允许降级的核心场景（如支付扣款失败需回滚），使用受检异常强制调用方处理。

Q2：受检异常在调用链中传递有什么问题？

回答要点：受检异常要求每层都声明 throws，底层修改异常类型会导致上层全部修改——破坏封装。Spring 将检查的 SQLException 包装为运行时 DataAccessException 就是解决这个问题。

Q3：什么时候适合用受检异常？

回答要点：当调用方有明确可执行的恢复策略时。例如连接池获取连接失败，调用方可以重试或降级；文件不存在时用户可以重新选择文件。如果调用方什么都做不了，请用运行时异常。
