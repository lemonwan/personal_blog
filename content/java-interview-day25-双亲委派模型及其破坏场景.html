核心概念：什么是双亲委派模型

双亲委派模型（Parent Delegation Model）是 Java 类加载器的默认协作机制：当一个 ClassLoader 收到类加载请求时，它通常不会先自己加载，而是先把请求委派给父加载器；只有父加载器无法完成加载时，子加载器才尝试自己加载。

  • Bootstrap ClassLoader：启动类加载器，主要加载 JDK 核心类库，如 java.lang.String、java.util.*。
  • Extension / Platform ClassLoader：Java 8 中常称扩展类加载器；Java 9+ 对应 Platform ClassLoader，加载平台相关类。
  • Application ClassLoader：应用类加载器，加载 classpath/module path 下的业务类和第三方依赖。
  • 自定义 ClassLoader：框架、插件化、热部署、隔离加载等场景会使用。

面试中要注意：这里的“父加载器”并不一定是继承关系，而更多是 ClassLoader 内部维护的 parent 引用形成的层级委派关系。

⚙️

加载流程：loadClass 的典型步骤

  1. 先检查是否已经加载：调用 findLoadedClass(name)，避免重复加载同一个类。
  2. 委派父加载器加载：如果 parent 不为空，调用 parent.loadClass(name, false)；如果 parent 为空，则交给 Bootstrap ClassLoader。
  3. 父加载器失败后自己加载：父加载器抛出 ClassNotFoundException 后，当前加载器再调用 findClass(name)。
  4. 按需解析链接：如果 resolve=true，再调用 resolveClass(clazz) 完成符号引用解析。

简化伪代码：

protected Class<?> loadClass(String name, boolean resolve) {
    Class<?> c = findLoadedClass(name);
    if (c == null) {
        try {
            c = parent != null ? parent.loadClass(name, false)
                               : findBootstrapClassOrNull(name);
        } catch (ClassNotFoundException e) {
            c = findClass(name);
        }
    }
    if (resolve) resolveClass(c);
    return c;
}

因此，自定义类加载器通常建议重写 findClass，而不是直接重写 loadClass，除非你明确要改变委派顺序。

为什么需要双亲委派：安全、唯一、隔离

双亲委派模型的价值主要体现在三个方面：

  • 核心类库安全：业务代码即使定义了一个 java.lang.String，也会优先由 Bootstrap ClassLoader 加载真正的 JDK String，避免核心类被恶意替换。
  • 类的唯一性：在同一个委派体系中，越基础的类越由上层加载器统一加载，减少同名类被重复定义导致的类型混乱。
  • 职责分层清晰：JDK 类、平台类、应用类分别由不同层级加载，降低类路径冲突概率。

需要牢记：JVM 判断两个类是否“相同”，不仅看全限定类名，还要看定义它的类加载器。同名 class 被不同 ClassLoader 加载后，在 JVM 看来可能是两个不同类型，强转时可能出现 ClassCastException。

⚖️

常见破坏场景：不是坏事，而是为了满足特殊需求

“破坏双亲委派”不是一定错误，很多框架是有意改变委派顺序来实现隔离、插件化或服务发现。

场景为什么要破坏典型例子
JDBC SPI核心类库需要反向加载厂商实现类DriverManager 借助线程上下文类加载器加载 MySQL Driver
Tomcat Web 应用隔离不同 WebApp 可以使用不同版本依赖WebAppClassLoader 优先加载自身 WEB-INF/lib/classes
OSGi / 插件化模块之间需要精细化依赖和版本隔离按 Bundle 维度加载类
热部署 / 动态加载卸载旧 ClassLoader，重新加载新 classSpring Boot DevTools、IDE 热加载、脚本引擎

面试回答时要强调：破坏通常发生在 loadClass 的委派顺序上，例如改为“子加载器优先”，或借助 Thread Context ClassLoader 让父级代码访问子级实现。

高频面试题与回答要点

Q1：双亲委派模型解决了什么问题？
答：保证核心类库安全，避免重复加载，维持类加载层次的稳定性。比如用户自定义 java.lang.String 不会覆盖 JDK 的 String。

Q2：ClassLoader 的父子关系是继承吗？
答：不是。主要是组合关系，ClassLoader 内部有一个 parent 引用。

Q3：为什么 JDBC 会破坏双亲委派？
答：DriverManager 属于 JDK 核心类，由上层加载器加载；但数据库驱动实现位于应用 classpath，由应用类加载器加载。父加载器默认看不到子加载器的类，所以需要线程上下文类加载器反向加载厂商驱动。

Q4：两个类全限定名一样，是否一定是同一个类？
答：不一定。还要看是否由同一个 ClassLoader 定义。不同加载器加载的同名类在 JVM 中可视为不同类型。

Q5：自定义 ClassLoader 应该重写哪个方法？
答：常规情况下重写 findClass，保留 loadClass 的双亲委派逻辑；只有需要子优先或特殊隔离时才重写 loadClass。

实际运用：Tomcat 为什么能隔离不同 Web 应用

Tomcat 中每个 Web 应用通常有自己的 WebAppClassLoader。这样 App A 和 App B 即使引用不同版本的同一个 jar，也可以在各自的加载空间内独立存在。

  • 共享类：Tomcat 自身类、JDK 类由公共或更上层加载器加载。
  • 应用私有类：WEB-INF/classes 和 WEB-INF/lib 由 WebAppClassLoader 加载。
  • 隔离收益：一个应用升级 Guava、Jackson、Spring 版本，不一定影响另一个应用。

但这也带来风险：如果静态变量、线程、ThreadLocal、JDBC Driver 等资源持有了旧 WebAppClassLoader 引用，应用停止后 class 元数据仍不能释放，就可能出现类加载器泄漏。

⚠️

踩坑经验：类冲突、ClassCastException 与热部署泄漏

  • 同名类冲突：排查时不要只看包名类名，还要看是哪个 jar、哪个 ClassLoader 加载的。可用 -verbose:class 或 Arthas sc -d 类名 查看。
  • 莫名 ClassCastException：常见原因是接口或 DTO 被两个不同 ClassLoader 分别加载，虽然全限定名一样，但 JVM 认为不是同一个类型。
  • 热部署内存泄漏：线程池、Timer、ThreadLocal、JDBC Driver、日志框架回调等如果不清理，会持有旧 ClassLoader，导致 Metaspace 增长。
  • 重写 loadClass 风险：子优先加载容易覆盖父级依赖，造成框架内部类型不一致；除非明确知道隔离边界，否则优先重写 findClass。

生产建议：对插件化或热部署系统，要设计明确的生命周期：加载、启动、停止、资源释放、ClassLoader 丢弃，并配合监控 Metaspace 和线程泄漏。

一页速记

  • 双亲委派：先父后子，父加载不了，子再加载。
  • 目的：安全、避免重复、层次清晰。
  • 类唯一性 = 全限定类名 + 定义它的 ClassLoader。
  • 常规自定义加载器：重写 findClass。
  • 破坏场景：JDBC SPI、Tomcat、OSGi、热部署、插件化。
  • 线程上下文类加载器：解决父加载器代码需要加载子加载器实现的问题。
  • 排查类加载问题：看 jar 来源、ClassLoader、是否重复依赖、是否热部署残留。
