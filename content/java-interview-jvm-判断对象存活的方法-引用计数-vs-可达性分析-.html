Day 20 — 判断对象存活的方法（引用计数 vs 可达性分析）

模块：JVM · 2. GC 机制
20 判断对象存活的方法（引用计数 vs 可达性分析）

📖核心概念：为什么需要判断对象存活？

垃圾收集器在回收内存之前，必须判断哪些对象还“存活”，哪些对象已经“死亡”。这是整个GC机制的起点。

Java中主要有两种判断存活的方法：

  • 引用计数法（Reference Counting）— 给对象添加一个引用计数器，有地方引用就+1，引用失效就-1
  • 可达性分析算法（Reachability Analysis）— Java 虚拟机采用的方法，从 GC Roots 出发，沿引用链向下搜索

⚖️引用计数法 — 原理与缺陷

原理：

  • 对象每被引用一次，计数器 +1
  • 引用失效时，计数器 -1
  • 计数器为 0 的对象即可被回收

缺陷（导致 Java 弃用此方法）：

⚠️ 循环引用问题
两个对象互相引用，即使它们已经不可达，计数器也不为0，无法回收。

// 伪代码示意循环引用问题 class A { B ref; } class B { A ref; } A a = new A(); B b = new B(); a.ref = b; // a计数器=1, b计数器=1 b.ref = a; // a计数器=2, b计数器=2 // a和b都指向空了，但计数器都不为0 // 对象永远无法被回收 → 内存泄漏!

Python 曾经使用引用计数（+GC周期性的循环回收），而 CPython 的 GC 就是为了解决循环引用问题。

🔒可达性分析算法 — Java的选择

从一组称为 "GC Roots" 的对象（根节点）出发，沿引用链向下搜索。

如果某个对象到 GC Roots 之间没有任何引用链相连（即从 GC Roots 到这个对象不可达），则判定该对象为可回收对象。

📖 引用链示意图
GC Roots → ObjectA → ObjectB → ObjectC → ObjectD(不可达,可回收)

可以作为 GC Roots 的对象包括：

  • 虚拟机栈（栈帧中的本地变量表）中引用的对象
  • 方法区中类静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中 JNI 引用的对象

🔬四种引用类型 — Java 精细化管理

Java 提供四种引用强度，决定对象被 GC 回收的时机：

引用类型说明回收时机
强引用正常创建的对象引用，如 Object o = new Object()永远不会被回收（除非引用断开）
软引用SoftReference，内存足够时不回收内存不足（OOM之前）时回收
弱引用WeakReferenceGC 时立即回收
虚引用PhantomReference，必须与引用队列联合使用无论何时，GC 时回收；主要用于对象被回收时的通知
// 强引用 — 最常见的引用方式 String s = new String("hello"); // 软引用 — 适合做缓存 SoftReference<byte[]> cache = new SoftReference<>(largeBytes); // 弱引用 — 防止内存泄漏 WeakReference<String> weak = new WeakReference<>(s); // 虚引用 — 对象回收通知 ReferenceQueue<Object> queue = new ReferenceQueue<>(); PhantomReference<Object> pr = new PhantomReference<>(obj, queue);

⚖️引用计数 vs 可达性分析 — 对比总结

维度引用计数法可达性分析法
回收准确性❌ 循环引用导致内存泄漏✅ 能正确处理循环引用
性能开销✅ 实时性高⚠️ 需要全局停顿
对象存活判定计数器为0即可判定需要遍历所有GC Roots
引用类型管理不支持细粒度支持强/软/弱/虚四种引用
Java是否采用❌ 未采用✅ HotSpot 采用

💡 面试要点
面试官可能会追问效率问题。答：现代 JVM 采用增量更新和安全点等机制优化，且 GC 通常只扫描存活对象（对象分代假设）。

🎤面试高频题

Q1: 为什么 Java 使用可达性分析而不是引用计数？

答：核心原因是循环引用问题。引用计数法无法处理对象互相引用的场景，会导致内存泄漏。

Q2: 什么是 GC Roots？

答：GC Roots 是可达性分析的起点对象，包括虚拟机栈引用、方法区静态属性、常量、JNI 本地方法栈引用等。

Q3: 四种引用类型在什么场景下使用？

答：软引用用于缓存（内存充足不回收，不足时回收）；弱引用用于防止内存泄漏；虚引用用于对象回收通知。

Q4: ThreadLocal 与弱引用有什么关系？

答：ThreadLocal 的 key 是弱引用，但 value 是强引用。如果线程不终止会导致 value 无法回收 — 所以必须手动 remove。
