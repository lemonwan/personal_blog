核心概念：CPU 100% 本质上在找“谁在烧 CPU”

线上 Java 服务 CPU 飙高时，排查目标不是先重启，而是快速定位：

  • 哪个进程占用 CPU 最高；
  • 进程里的哪个线程占用 CPU 最高；
  • 这个线程对应到 JVM 中的哪段 Java 调用栈；
  • 是否需要进一步结合 jmap、GC 日志、业务日志判断是死循环、锁竞争、频繁 GC、正则灾难还是流量突增。

经典链路可以记成：top 找进程 → top -Hp 找线程 → printf 转十六进制 → jstack 查线程栈 → jmap/日志辅助判断。

💻

标准排查流程：top → jstack → jmap

  1. 定位高 CPU 进程
    top 查看 %CPU 最高的 Java 进程，记录 PID，例如 12345。
  2. 定位进程内高 CPU 线程
    top -Hp 12345，找到最耗 CPU 的线程 ID，记为十进制 TID，例如 12372。
  3. 线程 ID 转十六进制
    JVM 线程栈里的 nid 是十六进制：printf '%x\n' 12372，得到如 3054。
  4. 导出线程栈并定位 nid
    jstack -l 12345 > /tmp/jstack_12345.log，搜索 nid=0x3054，查看该线程当前正在执行的方法栈。
  5. 必要时连续抓取多次
    间隔 5~10 秒执行 3 次 jstack。如果同一线程一直停在同一业务方法，常见是死循环、超大集合遍历、复杂正则、JSON 序列化、加解密压缩等 CPU 密集逻辑。
  6. 结合 jmap/GC 判断内存与 GC 问题
    若 jstack 发现大量 GC 线程活跃，或应用线程频繁分配对象，可用 jstat -gcutil PID 1000 10、jmap -histo:live PID、Heap Dump 辅助分析。
🔍

线程栈怎么看：RUNNABLE 不一定就是问题根因

面试中常见追问：看到线程是 RUNNABLE 就一定说明它在占 CPU 吗？答案是不一定。

  • RUNNABLE 可能真的在运行，例如死循环、Hash 冲突导致长时间遍历、复杂计算。
  • RUNNABLE 也可能阻塞在 native IO 上，例如 socket read、文件 IO，此时 CPU 未必高。
  • BLOCKED 通常表示等待进入 synchronized 监视器，重点看持有锁的线程。
  • WAITING/TIMED_WAITING 常见于线程池取任务、sleep、park，一般不是 CPU 100% 的直接原因。

判断关键是：高 CPU 的 TID 对应的栈顶方法是什么，以及多次采样是否持续停留在同一热点栈。

常见根因分类

现象典型原因排查要点
单个业务线程 CPU 高死循环、递归、复杂正则、超大集合处理看高 CPU 线程栈顶业务代码
多个业务线程都高流量突增、热点接口、批量任务并发过高结合接口 QPS、慢日志、线程池队列
GC 线程活跃对象创建过快、内存泄漏、堆太小看 jstat、GC 日志、jmap -histo
大量线程 BLOCKED锁竞争严重看 jstack -l 中 locked/waiting to lock
CPU 高但 jstack 不明显JNI/native、系统调用、容器限额结合 perf、容器 CPU limit、宿主机指标

面试真题：线上 CPU 100%，你怎么处理？

推荐回答：

  1. 先保护现场，不盲目重启；如果已经影响核心链路，先限流、摘流量或扩容止血。
  2. 用 top 找到高 CPU Java 进程 PID。
  3. 用 top -Hp PID 找到进程内高 CPU 线程 TID。
  4. 将 TID 转成十六进制，用 jstack -l PID 搜索 nid=0x...，定位线程调用栈。
  5. 连续抓 3 次线程栈，确认热点栈是否稳定，避免偶然采样误判。
  6. 如果是业务代码，结合日志、入参、最近发布、流量变化定位；如果是 GC 线程高，继续看 GC 日志、jstat、jmap。
  7. 最后给出修复方案，例如修复死循环、优化算法、增加索引/缓存、控制线程池并发、调整 JVM 参数或回滚版本。

加分点：强调“先止血、再定位、保留现场、复盘根因”，这比只背命令更符合真实生产处理方式。

⚠️

踩坑经验：这些操作要谨慎

  • 不要一上来 kill -9。重启会丢失现场，除非故障已严重影响业务且没有摘流量条件。
  • Heap Dump 可能造成 STW 或磁盘打满。生产执行 jmap -dump:live 前要确认磁盘空间和业务影响。
  • 线程 ID 进制容易搞错。top -Hp 是十进制，jstack nid 是十六进制。
  • 只看一次 jstack 容易误判。线程栈是瞬时快照，应该多次采样比对。
  • 容器环境要看 CPU limit。K8s 中 1 核 limit 下 Java 进程显示 100% 可能只是打满容器配额，不等于宿主机满载。
❓

速记口诀

CPU 100% 排查口诀：

先止血，保现场；top 找进程，top -Hp 找线程；十进制转十六进制；jstack 找 nid；多次采样看热点；GC 高再接 jstat/jmap；业务栈回到代码和最近变更。

面试表达时，可以把命令和排障思路结合起来讲：命令解决“定位”，日志和代码解决“根因”，限流/回滚/扩容解决“止血”。
