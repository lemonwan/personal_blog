核心概念：GC 日志到底看什么

GC 日志是 JVM 在发生垃圾回收时输出的结构化运行记录，用来回答三个关键问题：什么时候 GC、为什么 GC、回收后效果如何。

  • 触发原因：如 Allocation Failure、Metadata GC Threshold、System.gc()、G1 的 G1 Evacuation Pause。
  • GC 类型：Young GC、Mixed GC、Full GC，不同类型代表不同的内存压力和停顿风险。
  • 内存变化：重点看 GC 前后堆、年轻代、老年代/Metaspace 的使用量变化。
  • 耗时停顿：关注 user/sys/real 或 Pause Time，其中 real 更接近业务线程实际停顿感知。
面试中不要只说“看 GC 次数和时间”，更要说明：通过日志判断对象晋升、老年代增长、Full GC 触发原因和是否存在内存泄漏趋势。
💻

常用开启参数：JDK 8 与 JDK 9+ 写法

JDK 8 常见参数：

-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-XX:+PrintGCTimeStamps
-Xloggc:/data/logs/gc.log
-XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFiles=10
-XX:GCLogFileSize=100M

JDK 9+ 统一日志参数：

-Xlog:gc*,safepoint:file=/data/logs/gc.log:time,uptime,level,tags:filecount=10,filesize=100M
  • gc*：输出 GC 相关详细信息。
  • safepoint：辅助判断应用停顿是否只由 GC 引起。
  • filecount/filesize：日志滚动，避免磁盘被打满。
生产建议：GC 日志默认打开并配置滚动，路径放在独立日志目录，配合应用发布时间点和监控指标一起分析。

日志字段解读：一条 Young GC 怎么读

典型 JDK 8 Parallel/CMS 风格日志：

2026-06-25T08:00:01.123+0800: 12345.678: [GC (Allocation Failure)
 [PSYoungGen: 65536K->8192K(76288K)] 120000K->70000K(251392K), 0.0356789 secs]
 [Times: user=0.08 sys=0.01, real=0.04 secs]
字段含义面试解读
Allocation FailureEden 分配失败触发 Young GC高频出现通常说明分配速率高或年轻代偏小
65536K->8192K(76288K)年轻代 GC 前、后、总容量回收后仍很高，可能存活对象多，会加速晋升
120000K->70000K整个堆 GC 前后变化如果回收效果差，要关注老年代增长
real=0.04 secs真实停顿时间业务延迟最关注 real 时间

如果 Young GC 后堆总使用量下降不明显，同时老年代持续上涨，就要怀疑大对象、长生命周期缓存或对象晋升过快。

G1 GC 日志重点：Young、Mixed、Humongous

G1 的日志更强调 Region、停顿预测和阶段耗时。常见事件包括：

  • G1 Evacuation Pause：疏散暂停，复制存活对象到新的 Region，可能是 Young 或 Mixed。
  • Concurrent Mark Cycle：并发标记周期，用于识别老年代 Region 的回收价值。
  • Mixed GC：既回收年轻代，也选择部分收益高的老年代 Region。
  • Humongous Allocation：大对象分配，超过 Region 一半会按巨型对象处理，容易导致空间碎片和 Full GC。
G1 分析口诀：看暂停目标是否达标、Mixed GC 是否正常出现、老年代是否持续增长、Humongous 对象是否过多、是否退化为 Full GC。

若日志中频繁出现 to-space exhausted、Evacuation Failure，说明疏散空间不足，可能需要增大堆、降低晋升压力或调整 InitiatingHeapOccupancyPercent。

💻

排查思路：从 GC 日志定位问题

  1. 先看现象：接口 RT 抖动、CPU 飙高、吞吐下降、服务假死还是 OOM。
  2. 看 GC 频率：Young GC 是否过于频繁；Full GC 是否频繁或停顿过长。
  3. 看回收效果：每次 GC 后堆是否能明显下降；老年代是否呈阶梯式上升且不回落。
  4. 结合线程与堆：GC 异常时配合 jstat、jmap -dump、MAT、jstack 分析对象来源和线程状态。
  5. 关联业务变更：检查版本发布、流量峰值、定时任务、缓存策略、批处理任务。
# 观察各区容量、使用率和 GC 次数
jstat -gcutil <pid> 1000 20

# 生成堆转储，配合 MAT 分析 Dominator Tree
jmap -dump:format=b,file=/tmp/heap.hprof <pid>

高频面试题与回答要点

Q1：GC 日志中 Full GC 很频繁，可能有哪些原因？

  • 老年代空间不足：对象存活时间长、缓存无边界、集合持续增长。
  • 大对象或批量对象直接进入老年代。
  • 元空间不足触发 Full GC，如动态代理/类加载过多。
  • 代码中显式调用 System.gc()。
  • G1 疏散失败、并发标记跟不上分配速度导致退化。

Q2：如何判断是否存在内存泄漏？

  • 观察多次 Full GC 后老年代最低水位是否持续升高。
  • GC 后堆无法回落，且伴随频繁 Full GC 或最终 OOM。
  • 抓取不同时间点 Heap Dump，对比对象数量、Retained Size 和 GC Roots 引用链。

Q3：GC 调优的目标是什么？ 不是“让 GC 次数最少”，而是在吞吐、停顿时间和内存成本之间取平衡，满足业务 SLA。

⚠️

踩坑经验：别被单条日志误导

  • 只看平均值不看分位数：一次几秒 Full GC 就可能造成大量请求超时。
  • 只看 GC 不看 Safepoint：应用停顿也可能来自偏向锁撤销、类重定义、代码反优化等安全点事件。
  • 盲目调大堆：能降低 GC 频率，但可能让 Full GC 停顿更长；先确认对象生命周期和泄漏问题。
  • 忽略日志滚动：GC 日志持续写入可能打满磁盘，导致服务异常。
  • 脱离业务上下文：高分配速率可能来自正常流量峰值，也可能来自某个批量接口或定时任务。
一句话总结：GC 日志分析的核心是“趋势 + 回收效果 + 触发原因 + 业务时间线”四者结合。
