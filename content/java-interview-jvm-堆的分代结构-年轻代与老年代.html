18 堆的分代结构：年轻代与老年代

📖堆的分代结构：年轻代与老年代

JVM 堆内存按照对象生命周期划分为年轻代（Young Generation）和老年代（Old Generation），这是「分代收集理论」的核心实现。

年轻代（Young Generation）

新创建的对象首先被分配在年轻代。年轻代进一步分为：

  • Eden 区：对象首次分配的地方，大多数对象在这里「朝生夕死」。
  • Survivor 区（From）：第一次Minor GC后存活下来的对象进入此区。
  • Survivor 区（To）：下一次Minor GC的目标区域。

年轻代默认占堆内存的 1/3，老年代占 2/3。可通过 -XX:NewRatio 调整比例。

老年代（Old Generation）

  • 存放长期存活的对象，如缓存对象、单例等。
  • 对象经过多次GC仍存活（默认 15次，可由 -XX:MaxTenuringThreshold 设定），会被晋升到老年代。
  • 大对象（超过Survivor区一半容量）可能直接分配在老年代（-XX:+PretenureSizeThreshold）。
  • 老年代满了触发 Full GC。
特性年轻代老年代
GC类型Minor GC（Young GC）Full GC
GC频率高（频繁回收）低（较少触发）
回收速度快慢
默认占比1/32/3

对象在年轻代的流转

对象创建 & 晋升流程： ┌──────────────────────────────────────────┐ │ 1. 新对象 → Eden区分配 │ │ 2. Minor GC：Eden存活对象 → Survivor(From) │ │ 3. 下次Minor GC：From → Survivor(To) │ │ 4. 反复交换，计数器+1 │ │ 5. 达到阈值 → 晋升老年代 │ │ 6. 大对象 / 年老对象 → 直接进老年代 │ └──────────────────────────────────────────┘

面试高频追问

  • 为什么分代收集？ — 不同生命周期对象采用不同GC策略，提高回收效率。年轻代GC频繁但快，老年代GC少但影响大。
  • Survivor区大小为何1:1？ — 避免内存碎片，From/To交替使用，确保存活的对象有空间可放。
  • 对象一定在Eden分配吗？ — 不一定。TLAB优先在Eden的线程私有区域分配；大对象和年老对象可能直接进入老年代。

🔬TLAB与指针碰撞/空闲列表

对象创建是 JVM 最频繁的操作之一，需要高效分配内存。

指针碰撞（Bump the Pointer）

当堆内存是规整的（所有已用内存连成一块），JVM 用一个指针做分界线：

  • 分配时把指针向空闲区域移动对象大小的距离。
  • 类似 malloc 的行为，速度极快。
  • 前提：使用 Serial / Parallel GC 等能确保堆规整的收集器。

空闲列表（Free List）

当堆内存不规整时（如 CMS/G1 收集器，内存碎片化），使用空闲列表：

  • JVM 维护一个记录空闲区域的列表。
  • 分配时从列表中找到足够大的空闲块。
  • 速度稍慢于指针碰撞，但更灵活。

TLAB（Thread Local Allocation Buffer）

核心问题：对象分配需要线程安全，但加锁会严重影响并发性能。

TLAB 是 Eden 区中的一块线程私有内存区域，每个线程有独立的分配缓冲区：

  • 优势：线程在 TLAB 内分配对象无需加锁，只需移动指针。
  • 触发方式：大多数对象优先尝试在 TLAB 分配；TLAB 满了或对象太大时，才在 Eden 全局区域分配并加锁。
  • 参数：-XX:+UseTLAB 开启（默认开启），-XX:TLABWasteTargetPercent 控制浪费比例。

面试重点

「对象是否一定在年轻代创建？」— 不一定。在 G1 中，TLAB 也是分代的；在 ZGC 中，所有对象分配在同一个逻辑堆，使用染色指针，不再区分年轻代/老年代的内存隔离。

🎤面试模拟

Q1：请描述 JVM 堆内存的分代结构。

参考回答：JVM 堆分为年轻代和老年代。年轻代又分为 Eden 区和两个 Survivor 区（From/To），默认比例为 8:1:1。新对象先在 Eden 区分配，经过 Minor GC 存活后进入 Survivor 区，多次 GC 仍存活的对象晋升到老年代。老年代存放长期存活的大对象。年轻代占总堆 1/3，老年代占 2/3。

Q2：对象创建时内存分配有哪几种方式？

参考回答：（1）指针碰撞 — 堆规整时使用，速度快，适用于 Serial/Parallel GC；（2）空闲列表 — 堆不规整时使用，适用于 CMS/G1 等收集器；（3）TLAB 线程私有的分配缓冲区，无需加锁，提高并发分配性能。

Q3：为什么 Survivor 区是 1:1 的两块？

参考回答：采用 From/To 双缓冲区设计，可以避免内存碎片，每次 Minor GC 时对象在两个 Survivor 区间来回复制。这样既保证了存活对象有足够的空间存放，又简化了内存管理。默认只使用其中一个 Survivor 区，另一个作为空备用区。

Q4：TLAB 有什么作用？为什么需要它？

参考回答：TLAB 是为了解决多线程并发分配内存的性能问题。如果不使用 TLAB，每次分配对象都需要对 Eden 区加锁同步。使用 TLAB 后，每个线程在私有缓冲区中分配对象无需加锁，显著提升了并发性能。当 TLAB 不足时才会退回到全局 Eden 分配。
